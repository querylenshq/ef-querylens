## ISSUE-02 · DbContext "pool" is a mutex-protected singleton, not a pool [HIGH]

**File:** `src/EFQueryLens.Core/Engine/QueryLensEngine.DbContextPool.cs`

**Problem:**
Each `PooledDbContext` is guarded by `new SemaphoreSlim(1, 1)` — one instance, one caller at a time. This serializes all translation requests for a given DbContext type. With 50 items in the translation queue, all 50 wait in line behind this semaphore. It is functionally a singleton with a mutex, not a pool.

**Current code (lines 29–35):**
```csharp
pooled = new PooledDbContext(
    poolKey,
    dbContextType.FullName!,
    instance,
    new SemaphoreSlim(1, 1),   // ← one slot = serialized
    strategy);
_dbContextPool[poolKey] = pooled;
```

**Fix Option A — Pool of N instances (recommended):**

Change `PooledDbContext` to hold a `ConcurrentQueue<object>` of instances and a `SemaphoreSlim(N, N)` where N is configurable (default 4):

```csharp
private sealed record PooledDbContextPool(
    string PoolKey,
    string DbContextTypeFullName,
    ConcurrentQueue<object> Instances,
    SemaphoreSlim Gate,          // SemaphoreSlim(N, N)
    string CreationStrategy,
    int PoolSize);
```

On `AcquireDbContextLeaseAsync`: `await gate.WaitAsync(ct)` then `instances.TryDequeue(out var instance)`.
On release: `instances.Enqueue(instance)` then `gate.Release()`.
On first acquire when queue is empty: create a new instance up to `PoolSize`.

Read pool size from env var `QUERYLENS_DBCONTEXT_POOL_SIZE` (default 4, max 16).

**Fix Option B — Rename for honesty (minimal change):**

If keeping single-instance is intentional (offline contexts are cheap to hold), rename to make the semantics explicit:

```csharp
// Rename the record
private sealed record SingletonDbContext(   // was: PooledDbContext
    string PoolKey,
    string DbContextTypeFullName,
    object Instance,
    SemaphoreSlim Gate,
    string CreationStrategy);

// Rename the dictionary
private readonly ConcurrentDictionary<string, SingletonDbContext> _dbContextSingletons = new(...);
```

And update the strategy value from `"pooled-reuse"` → `"singleton-reuse"` in all log messages and `TranslationMetadata.CreationStrategy`.

## ISSUE-03 · Global lock held during directory manifest scan [HIGH]

**File:** `src/EFQueryLens.Core/AssemblyContext/ShadowAssemblyCache.cs`
**Lines:** `44–85`

**Problem:**
`BuildManifest(sourceDir)` (which enumerates every file in the build output directory — potentially 500+ DLLs) runs inside `lock (_gate)`. This blocks all concurrent translation requests for the duration of the scan plus file copy.

**Current code:**
```csharp
lock (_gate)
{
    var manifest = BuildManifest(sourceDir);          // ← slow: FS enumeration
    var bundleKey = ComputeBundleKey(sourceDir, manifest);
    var bundlePath = Path.Combine(_bundleRoot, bundleKey);
    bundleAssemblyPath = Path.Combine(bundlePath, Path.GetFileName(fullSourcePath));

    if (File.Exists(bundleAssemblyPath))
    {
        TouchDirectory(bundlePath);
    }
    else
    {
        var stagingPath = ...;
        // ... file copy to staging ...     // ← also slow
        TryAtomicPromote(stagingPath, bundlePath);
    }
}
```

**Fix:**
Move manifest build and file copy outside the lock. Only hold the lock for the atomic promote check:

```csharp
public string ResolveOrCreateBundle(string sourceAssemblyPath)
{
    var fullSourcePath = Path.GetFullPath(sourceAssemblyPath);
    var sourceDir = Path.GetDirectoryName(fullSourcePath)
        ?? throw new InvalidOperationException($"Could not determine source directory for '{fullSourcePath}'.");

    if (!Directory.Exists(sourceDir))
        throw new DirectoryNotFoundException($"Source output directory not found: {sourceDir}");

    // ── Build manifest and compute key OUTSIDE the lock ──────────────────────
    var manifest = BuildManifest(sourceDir);
    var bundleKey = ComputeBundleKey(sourceDir, manifest);
    var bundlePath = Path.Combine(_bundleRoot, bundleKey);
    var bundleAssemblyPath = Path.Combine(bundlePath, Path.GetFileName(fullSourcePath));

    // Fast path: bundle already exists — just touch it.
    if (File.Exists(bundleAssemblyPath))
    {
        lock (_gate) { TouchDirectory(bundlePath); }
        ScheduleCleanupIfDue(force: false);
        return bundleAssemblyPath;
    }

    // Slow path: copy to staging outside the lock, then promote under lock.
    var stagingPath = Path.Combine(_stagingRoot, $"{bundleKey}-{Guid.NewGuid():N}");
    Directory.CreateDirectory(stagingPath);

    try
    {
        foreach (var entry in manifest)
        {
            var targetPath = Path.Combine(stagingPath, entry.RelativePath);
            var targetDir = Path.GetDirectoryName(targetPath);
            if (!string.IsNullOrWhiteSpace(targetDir))
                Directory.CreateDirectory(targetDir);
            File.Copy(entry.FullPath, targetPath, overwrite: true);
        }

        // ── Atomic promote INSIDE the lock ───────────────────────────────────
        lock (_gate)
        {
            // Re-check: another thread may have promoted while we were copying.
            if (!File.Exists(bundleAssemblyPath))
                TryAtomicPromote(stagingPath, bundlePath);
            TouchDirectory(bundlePath);
        }
    }
    finally
    {
        TryDeleteDirectory(stagingPath);
    }

    ScheduleCleanupIfDue(force: true);
    return bundleAssemblyPath;
}
```

## ISSUE-04 · `ChangeTracker.Clear()` via uncached reflection on every lease release [MEDIUM]

**File:** `src/EFQueryLens.Core/Engine/QueryLensEngine.DbContextPool.cs`
**Lines:** `91–104`

**Problem:**
`GetProperty` + `GetMethod` are called on every release. These are not cheap — each walks the full type hierarchy. Over hundreds of hover requests per session, this adds up.

**Current code:**
```csharp
private static void ClearChangeTracker(object dbContextInstance)
{
    var changeTrackerProp = dbContextInstance.GetType()
        .GetProperty("ChangeTracker", BindingFlags.Public | BindingFlags.Instance);

    if (changeTrackerProp?.GetValue(dbContextInstance) is not { } changeTracker)
        return;

    var clearMethod = changeTracker.GetType()
        .GetMethod("Clear", BindingFlags.Public | BindingFlags.Instance);
    clearMethod?.Invoke(changeTracker, null);
}
```

**Fix:**
Cache the `MethodInfo` per DbContext type using a static `ConcurrentDictionary`:

```csharp
// Add to the class-level fields in QueryLensEngine.cs:
private static readonly ConcurrentDictionary<Type, Action<object>?> s_clearChangeTrackerCache = new();

private static void ClearChangeTracker(object dbContextInstance)
{
    var type = dbContextInstance.GetType();
    var clearDelegate = s_clearChangeTrackerCache.GetOrAdd(type, static t =>
    {
        var changeTrackerProp = t.GetProperty("ChangeTracker", BindingFlags.Public | BindingFlags.Instance);
        if (changeTrackerProp is null) return null;

        var clearMethod = changeTrackerProp.PropertyType
            .GetMethod("Clear", BindingFlags.Public | BindingFlags.Instance);
        if (clearMethod is null) return null;

        // Build a reusable delegate to avoid repeated reflection invocations.
        return instance =>
        {
            var changeTracker = changeTrackerProp.GetValue(instance);
            if (changeTracker is not null)
                clearMethod.Invoke(changeTracker, null);
        };
    });

    clearDelegate?.Invoke(dbContextInstance);
}
```

**Also note:** `ChangeTracker.Clear()` only resets tracked entities. It does not reset query caches, model caches, or state set in `OnConfiguring`. If a DbContext instance has been misconfigured by user factory code, pooling it will carry that state across requests. Consider adding a comment in `PooledDbContext` that documents this known limitation.

## ISSUE-05 · `_alcContextGates` dictionary never pruned [MEDIUM]

**File:** `src/EFQueryLens.Core/Engine/QueryLensEngine.cs`
**Lines:** `70–71`

**Problem:**
A gate `object` is added to `_alcContextGates` for every unique assembly path that is ever translated. Entries are never removed. In a long-lived LSP session where the user renames a project or changes output paths, old entries accumulate.

**Current code:**
```csharp
private readonly ConcurrentDictionary<string, object> _alcContextGates = new(
    StringComparer.OrdinalIgnoreCase);

// In GetOrRefreshContext:
var gate = _alcContextGates.GetOrAdd(sourceAssemblyPath, static _ => new object());
```

**Fix:**
Remove the gate when the ALC cache entry for that path is also removed. In `ReleaseCachedContextAsync`, after the ALC is disposed:

```csharp
private async ValueTask ReleaseCachedContextAsync(CachedAssemblyContext entry, string reason)
{
    _evaluator.InvalidateMetadataRefCache(entry.ShadowAssemblyPath);
    await EvictPooledDbContextsForAssemblyAsync(entry.SourceAssemblyPath);
    entry.Context.Dispose();
    ForceUnloadCollection(aggressive: string.Equals(reason, "dispose", StringComparison.Ordinal));

    // ── NEW: prune the gate if no cache entry references this path anymore ───
    if (!_alcCache.ContainsKey(entry.SourceAssemblyPath))
        _alcContextGates.TryRemove(entry.SourceAssemblyPath, out _);

    LogDebug($"alc-release reason={reason} source={entry.SourceAssemblyPath} shadow={entry.ShadowAssemblyPath}");
}
```

## ISSUE-06 · `_namespaceTypeIndexCache.Clear()` nukes all projects on any rebuild [MEDIUM]

**File:** `src/EFQueryLens.Core/Scripting/Evaluation/QueryEvaluator.cs`
**Lines:** `93–94`

**Problem:**
When any assembly is invalidated (e.g., ProjectA rebuilds), the entire namespace/type index cache is wiped — including the cached index for ProjectB, ProjectC, etc. The re-scan is expensive on large projects.

**Current code:**
```csharp
public void InvalidateMetadataRefCache(string assemblyPath)
{
    // ...
    _namespaceTypeIndexCache.Clear();    // ← nukes all projects
}
```

**Fix:**
The namespace/type index is keyed by `assemblySetHash`. Remove only the entries whose key contains the invalidated assembly's path as a component. Since the hash is opaque, the cleanest approach is to store the assembly path alongside the entry and filter on it:

```csharp
// Change NamespaceTypeIndexEntry to include the source assembly path:
private sealed record NamespaceTypeIndexEntry(
    string SourceAssemblyPath,    // ← ADD this field
    HashSet<string> Namespaces,
    HashSet<string> Types,
    long LastAccessTicks);

// In GetOrBuildNamespaceTypeIndex, store the path:
_namespaceTypeIndexCache[assemblySetHash] = new NamespaceTypeIndexEntry(
    normalized,                   // ← assembly path
    result.Namespaces,
    result.Types,
    GetUtcNowTicks());

// In InvalidateMetadataRefCache, filter instead of clearing all:
var normalized = Path.GetFullPath(assemblyPath);
foreach (var key in _namespaceTypeIndexCache
             .Where(kvp => string.Equals(
                 kvp.Value.SourceAssemblyPath, normalized, StringComparison.OrdinalIgnoreCase))
             .Select(kvp => kvp.Key)
             .ToList())
{
    _namespaceTypeIndexCache.TryRemove(key, out _);
}
```

## ISSUE-07 · `double` accumulation for millisecond rolling metrics [MEDIUM]

**File:** `src/EFQueryLens.Daemon/TranslationMetrics.cs`
**Lines:** `12–14`

**Problem:**
`double SumMs` accumulates integer millisecond values. Floating-point addition is not associative — after thousands of samples in a long-running LSP session, the running sum drifts. Since all inputs are whole-number milliseconds, `long` is both correct and faster.

**Current code:**
```csharp
private sealed class RollingMetricState
{
    public readonly object SyncRoot = new();
    public readonly Queue<long> Samples = new();
    public double SumMs;          // ← should be long
    public int TotalSampleCount;
    public long LastSampleMs;
}
```

**Fix:**
```csharp
private sealed class RollingMetricState
{
    public readonly object SyncRoot = new();
    public readonly Queue<long> Samples = new();
    public long SumMs;            // ← long: no floating-point drift
    public int TotalSampleCount;
    public long LastSampleMs;
}
```

Update `GetAverageMs` return type remains `double` (for the division), but the accumulation is exact:
```csharp
public double GetAverageMs(string contextName)
{
    // ...
    return (double)state.SumMs / state.Samples.Count;  // division produces double
}
```

## ISSUE-08 · Roslyn eval assemblies compiled at `OptimizationLevel.Release` [LOW]

**File:** `src/EFQueryLens.Core/Scripting/Evaluation/QueryEvaluator.cs`

**Problem:**

```csharp

    new(OutputKind.DynamicallyLinkedLibrary,
```

## ISSUE-11 · Record inheritance on `ExplainResult` breaks value semantics [LOW]

**File:** `src/EFQueryLens.Core/Contracts/Explain/ExplainResult.cs`

**Problem:**
Record inheritance makes `with`-expression cloning lossy (cloning through the base type drops derived fields) and breaks equality contracts in unexpected ways. It also forces `ExplainResult` to duplicate all `QueryTranslationResult` properties in its `with` expressions.

**Current code:**
```csharp
public sealed record ExplainResult : QueryTranslationResult
{
    public ExplainNode? Plan { get; init; }
    public bool IsActualExecution { get; init; }
    public string? ServerVersion { get; init; }
```

**Fix — use composition:**
```csharp
public sealed record ExplainResult
{
    public required QueryTranslationResult Translation { get; init; }
    public ExplainNode? Plan { get; init; }
    public bool IsActualExecution { get; init; }
    public string? ServerVersion { get; init; }

    public bool Success => Translation.Success;
    public string? Sql => Translation.Sql;
    public string? ErrorMessage => Translation.ErrorMessage;
    public TranslationMetadata Metadata => Translation.Metadata;
```

Update all construction sites that do `new ExplainResult { Success = ..., Sql = ..., Plan = ... }` to `new ExplainResult { Translation = new QueryTranslationResult { ... }, Plan = ... }`.

## ISSUE-12 · `IDesignTimeDbContextFactory<T>` already exists — no need for a custom factory interface [MEDIUM]

**File:** `src/EFQueryLens.Core/Contracts/IQueryLensDbContextFactory.cs`

**Problem:**
`IQueryLensDbContextFactory<T>` is discovered via duck-typed full-name reflection. Users must learn a new interface name. EF Core already has `IDesignTimeDbContextFactory<T>` (`Microsoft.EntityFrameworkCore.Design`) that users implement for EF migrations — and its `CreateDbContext(string[] args)` signature is almost identical to the custom factory's intent.

**Fix:**
In `DesignTimeDbContextFactory.cs`, add a second discovery pass that checks for EF Core's native interface **before** requiring the custom one:

```csharp
internal static class DesignTimeDbContextFactory
{
    private const string QueryLensInterfaceName =
        "EFQueryLens.Core.IQueryLensDbContextFactory`1";

    // EF Core's own design-time factory — used by migrations, discovered for free.
    private const string EfDesignTimeInterfaceName =
        "Microsoft.EntityFrameworkCore.Design.IDesignTimeDbContextFactory`1";

    public static (object Instance, string Strategy) TryCreate(
        Type dbContextType,
        IEnumerable<Assembly> assemblies)
    {
        // 1. Try QueryLens-specific factory first (explicit opt-in, no args needed).
        var qlResult = TryCreateQueryLensFactory(dbContextType, assemblies);
        if (qlResult.HasValue)
            return (qlResult.Value.Instance, "querylens-factory");

        // 2. Fall back to EF Core IDesignTimeDbContextFactory<T> (already implemented
        //    by users who run migrations — zero extra setup required).
        var efResult = TryCreateEfDesignTimeFactory(dbContextType, assemblies);
        if (efResult.HasValue)
            return (efResult.Value.Instance, "ef-design-time-factory");

        throw new InvalidOperationException(
            $"No factory found for '{dbContextType.Name}'. " +
            "Implement either IQueryLensDbContextFactory<T> or " +
            "IDesignTimeDbContextFactory<T> in your startup project.");
    }

    private static (object Instance, Type ContextType)? TryCreateEfDesignTimeFactory(
        Type dbContextType, IEnumerable<Assembly> assemblies)
    {
        var genericName = $"{EfDesignTimeInterfaceName}[[{dbContextType.AssemblyQualifiedName}]]";
        // ... same reflection discovery pattern as TryCreateQueryLensFactory
    }
}
```

## ISSUE-13 · LINQ property extraction uses fragile string parsing when Roslyn is available [MEDIUM]

**File:** `src/EFQueryLens.Core/AssemblyContext/ProjectAssemblyContext.DbContextDiscovery.cs`
**Lines:** `142–163`

**Problem:**
`ExtractFirstPropertyAccess` uses `IndexOf('.')` and a character walk. It fails silently for:
- `(db).Orders.Where(...)` — parenthesized context
- `db?.Orders.Where(...)` — null-conditional access
- `/* comment */\ndb.Orders` — leading comments
- `db .Orders` — space before dot

`Microsoft.CodeAnalysis.CSharp` is **already referenced by Core** so there is no additional dependency cost.

**Fix:**
```csharp
private static string? ExtractFirstPropertyAccess(string expression)
{
    // Use Roslyn to parse the expression properly instead of fragile string scanning.
    var tree = CSharpSyntaxTree.ParseText(
        expression,
        CSharpParseOptions.Default.WithKind(SourceCodeKind.Script));

    var root = tree.GetRoot();

    // Walk the tree looking for the first MemberAccessExpressionSyntax.
    var memberAccess = root.DescendantNodes()
        .OfType<Microsoft.CodeAnalysis.CSharp.Syntax.MemberAccessExpressionSyntax>()
        .FirstOrDefault();

    return memberAccess?.Name.Identifier.Text;
}
```

Add `using Microsoft.CodeAnalysis.CSharp;` and `using Microsoft.CodeAnalysis.CSharp.Syntax;` at the top of the file.

## ISSUE-16 · Shadow cache defaults to `%TEMP%` with no user override [LOW]

**File:** `src/EFQueryLens.Core/AssemblyContext/ShadowAssemblyCache.cs`
**Lines:** `21–23`

**Problem:**
`Path.GetTempPath()` on Windows resolves to `C:\Users\<user>\AppData\Local\Temp`. With a 5 GB soft limit, this can pressure the OS drive. Users on space-constrained machines or with separate data drives have no way to redirect the cache.

**Current code:**
```csharp
_root = Path.Combine(Path.GetTempPath(), "EFQueryLens", "shadow");
```

**Fix:**
```csharp
private static string ResolveShadowRoot()
{
    // Allow explicit override via environment variable.
    var envOverride = Environment.GetEnvironmentVariable("QUERYLENS_SHADOW_ROOT");
    if (!string.IsNullOrWhiteSpace(envOverride))
        return envOverride.Trim();

    // Default: LocalApplicationData is more appropriate than Temp for persistent cache data.
    // Windows: C:\Users\<user>\AppData\Local\EFQueryLens\shadow
    // Linux/macOS: ~/.local/share/EFQueryLens/shadow
    return Path.Combine(
        Environment.GetFolderPath(Environment.SpecialFolder.LocalApplicationData),
        "EFQueryLens", "shadow");
}

public ShadowAssemblyCache(bool debugEnabled)
{
    _debugEnabled = debugEnabled;
    _root = ResolveShadowRoot();
    _bundleRoot = Path.Combine(_root, "bundles");
    _stagingRoot = Path.Combine(_root, "staging");

    Directory.CreateDirectory(_bundleRoot);
    Directory.CreateDirectory(_stagingRoot);
}
```

Document `QUERYLENS_SHADOW_ROOT` in `docs/cli-reference.md` under environment variables.
